## 1. 抽取流程

force_json 模式下你**只能输出最终 JSON**，不得吐出思考文本。但下笔前必须在心里先完成“本{{TURN_UNIT}}进展 → 盘面校验”的阅读顺序，漏读会导致把旧闻当新事、把背景当结果。

### 1.1 先读本{{TURN_UNIT}}进展

所有模块都先通读本{{TURN_UNIT}}奏章各模块，建立“本{{TURN_UNIT}}到底发生了什么”的时间线

#### 1.1.1 诏书核销

读本{{TURN_UNIT}}诏书逐条落地情况：力度（投了多少银/人/权限）、阻力、结果三态。**核销章每条在叙事句末写 `（结果：完成/进行中/中断）`，不是 `（进展：档位）`**——这是诏书核销专用的结果词（落在该条叙事末尾、不在标题上），照它判该条去向：
- `完成`＝一锤子事（抄家/人事任免/军调/钱粮调度等）当{{TURN_UNIT}}了结，**不涉长期局势，不进 `新立局势`**。
- `进行中`＝长期事，**必有对应局势在第二章「长期局势」承接**（开头标 `#编号` 或 `新立『标题』`）。该条的 issue 进度（`进度增量`）/是否新立一律**去第二章那条读**——核销章只表明这道旨意本{{TURN_UNIT}}在推进，不在这里直接给 issue 增量。
- `中断`＝没办成/搁置/折损；命中在办局势则在第二章作负向推进，否则不立条。

相关背景在 `decree_text` 与盘面表。

#### 1.1.2 指令核销

读本{{TURN_UNIT}}固定指令库的逐条落地情况。指令核销格式更固定：`指令标题 — 类型：类别；字段：key=value...；叙事（结果：完成/进行中/中断）`。

- 指令核销来源是 simulator_payload.structured_directives，独立于 `decree_text`。
- 抽取钱粮、军队、军备、人物等硬字段时，优先相信指令核销里的结构化字段。
- `完成`＝一锤子事当{{TURN_UNIT}}了结，不因指令本身新立局势.
- `进行中`＝仅 `tax_reform` / 财政科目改革可用，且必须在「长期局势」章用 `#编号` 或 `新立『财政科目改革：…』` 承接。其它固定指令只认 `完成` / `中断`。
- 若诏书核销与指令核销语义重复，只落库一次；以指令核销的结构化字段校准金额、对象、动作。

#### 1.1.3 长期局势

读在办事项本{{TURN_UNIT}}停在哪、推进到哪、是否新冒出情势。某条若写「详见诏书核销第N条」（指针去重），其本{{TURN_UNIT}}实际进展在诏书核销那条，回去读那条再判此 issue 进度；指针句里的进退方向词（续进/更远/恶化）是该 issue 本{{TURN_UNIT}}增量正负的依据。相关背景在 `active_issues` 与 `candidate_events`。

#### 1.1.4 密令动向

读密令本{{TURN_UNIT}}办到哪一步、是否有风声/风险、是否进入核议；相关背景在 `secret_orders`。

### 1.2 再读盘面校验

基线盘面在 system 的 simulator_payload 里。其中 buildings/departments/technologies/court_roster/armies/regions 等表以 **TSV 文本块**给出（块头 `## 表名`，块内首行 tab 分隔列名，其后每行一条记录按列名对位、空字段为空串）；departments=已设衙门、technologies=已解锁科技（空表只有表头＝尚未设立，立新的别与已有重名）；powers/factions/classes 等其余字段在「## 其余字段（JSON）」内。算 delta 一律「相对该 TSV/JSON 里的当前值」，按列名取字段，不凭叙事印象。

三栏之外的旧闻、背景、议论、历史锚点只能辅助理解，不得单独当成本{{TURN_UNIT}}新进展；盘面只用于校验对象存在、状态合法、增量相对值，不得替代本{{TURN_UNIT}}进展本身。

## 2. 模块化输出纪律

- 你只输出当前模块允许的顶层字段；不属于当前模块的字段一律不要输出。
- 当前模块要求的顶层字段都必须出现；无内容的字段填 `{}` 或 `[]`。
- 所有输出字段名尽量用中文；程序会把中文字段标准化为内部字段。
- 字段类型、增量/新值语义必须与总 extractor 契约一致。
- 所有数值增量必须是 integer。能从字符串转成数字也要直接输出数字，不要输出 `"5"`。
- 严禁使用契约外别名字段，例如 `amount` / `value` / `money` / `cost` / `change`。
